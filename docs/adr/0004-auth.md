# ADR-0004: Authentication — forms + API key + OIDC + trusted proxy

- Status: Accepted
- Date: 2025-05-01
- Deciders: Loom maintainers

## Context

The user spectrum runs from "single user on a Pi behind their router" to
"homelab with Authelia/Authentik SSO and a reverse proxy" to "small team
with Entra ID". One auth model must cover all of them without nagging
the simple case.

## Decision

Loom supports four auth modes, configurable independently and stackable:

1. **Forms login** — username + password (argon2id hashed) → HTTP-only,
   SameSite=Lax cookie. First-run wizard creates the initial admin.
2. **API keys** — long-lived per-user keys, scoped (read-only / read-write
   / admin), rotated via UI/CLI. Required for downstream integrations
   (Overseerr/Bazarr/Notifiarr) that don't speak OIDC.
3. **OIDC** — any compliant IdP (Authelia, Authentik, Keycloak, Entra,
   Google, GitHub). PKCE auth code flow; ID-token claims map to local
   roles via configurable mapping.
4. **Trusted reverse-proxy header auth** — when sat behind authelia/
   authentik in forward-auth mode. Header name and trusted CIDR are
   explicit; off by default to prevent header-spoofing.

Authorization is **role-based** (viewer, user, admin) with optional
fine-grained permissions for the plugin SDK.

## Consequences

### Positive
- New users get the simple cookie experience; advanced users plug into
  their existing SSO without code changes.
- API keys keep the entire downstream ecosystem working.
- Header-trust mode is opt-in and CIDR-gated, eliminating the most common
  reverse-proxy auth footgun.

### Negative / trade-offs
- Four auth modes are more code than one; we contain complexity in
  `internal/server/auth/` with a single `Authenticator` interface.
- Argon2id parameters need tuning per release (we benchmark in CI).

### Neutral
- Sessions stored in DB; revocable. No JWTs for user sessions (avoids
  premature complexity).

## Alternatives considered

- **Forms only** — excludes SSO users and downstream integrations.
- **OIDC only** — non-starter for first-run simplicity and for downstream
  apps without OIDC support.
- **JWT user sessions** — revocation pain; DB-backed sessions are fine
  at our scale.
